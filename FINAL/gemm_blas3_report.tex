\section*{Introduction}
\addcontentsline{toc}{section}{Introduction}
%%============================================================

This report documents the implementation and performance evaluation of a
single-precision BLAS library developed over six weeks using AI-assisted development.
The codebase was built iteratively through prompting with the Antigravity IDE
(Claude Sonnet~4.6 and Gemini Pro), with the objective of reaching hardware peak
performance on a 20-core Intel Xeon AVX2 workstation.
All implementations are benchmarked against OpenBLAS to assess where AI-generated
kernels are competitive, where they fall short, and why.

\subsection*{Scope of Development}
The project spans six weeks of systematic, iterative development, categorized as follows:
\begin{itemize}
  \item \textbf{Part~1 (Weeks 1--3 \& 6): GEMM and BLAS~3.} The core focus is SGEMM (Single-Precision General Matrix Multiply), implementing the sophisticated BLIS (BLAS-like Library Instantiation Software) 5-loop architecture. This part also covers autotuning infrastructures and derived BLAS~3 operations (\texttt{ssyrk, strsm, strmm}).
  \item \textbf{Part~2 (Week 4): BLAS~1.} Implementation of foundational vector-vector operations (\texttt{sscal, scopy, sswap, saxpy, sdot, snrm2, sasum, isamax, srot}), pushing single-thread and multi-thread memory bandwidth limits.
  \item \textbf{Part~3 (Week 5): BLAS~2.} Implementation of matrix-vector operations (\texttt{sgemv, ssymv, strmv, strsv, sger, ssyr, ssyr2}), focusing on cache-friendly access patterns and algorithmic optimizations like loop fusion.
  \item \textbf{Part~4: Testing and Agentic Optimization.} Development of the rigorous numerical validation testbench and the autonomous parameter-tuning agent loop.
\end{itemize}

\subsection*{Hardware Architecture and Theoretical Limits}
Understanding the target hardware is critical for evaluating HPC software. The testbed is a 20-core Intel Xeon workstation (x86-64 architecture).

\paragraph{Compute Capabilities (AVX2 \& FMA).}
The processor supports Advanced Vector Extensions 2 (AVX2), featuring 256-bit wide vector registers. A single 256-bit register holds 8 single-precision floating-point numbers. The processor also supports Fused Multiply-Add (FMA) instructions, which perform $d = a \cdot b + c$ in a single hardware step, counting as 2 floating-point operations (FLOPs). 
A single core can issue two FMA instructions per clock cycle. At a boost frequency of 3.5\,GHz, the theoretical peak performance is:
\begin{equation}
\text{Peak}_{\text{core}} = 2 \text{ (instructions/cycle)} \times 8 \text{ (elements/vector)} \times 2 \text{ (FLOPs/FMA)} \times 3.5\text{e}9 \text{ (cycles/sec)} = 56 \text{ GFLOP/s}
\end{equation}
Fully utilized across all 20 cores, the machine theoretical peak is $1120 \text{ GFLOP/s}$ (1.12 TFLOP/s).

\paragraph{Memory Hierarchy and Bandwidth.}
The memory subsystem is equally critical. The CPU features three levels of cache:
\begin{itemize}
    \item \textbf{L1 Data Cache}: 32\,KB per core. Extremely fast (4-5 cycles), but small.
    \item \textbf{L2 Cache}: 256\,KB per core. Intermediate latency ($\sim$12-14 cycles).
    \item \textbf{L3 Cache}: $\approx$25\,MB, shared across all cores.
\end{itemize}
The main memory (RAM) utilizes DDR4-3200 modules. The theoretical maximum memory bandwidth for a single memory controller channel is roughly $25.6 \text{ GB/s}$. With a dual-channel configuration, the practical sustained bandwidth is approximately $45 \text{ GB/s}$.
The 25\,MB L3 cache, shared across all 20 cores, has a substantially higher aggregate internal bandwidth: when all cores access the L3 simultaneously, the internal ring/mesh interconnect can sustain $\approx 300$--$500$\,GB/s. This creates two distinct bandwidth regimes that appear throughout this report:
\begin{itemize}
    \item \textbf{DRAM-bound}: data exceeds L3 capacity ($> 25$\,MB); effective bandwidth $\approx 45$\,GB/s.
    \item \textbf{L3-bound}: data fits in L3 ($< 25$\,MB, e.g.\ vectors of $N = 2^{20}$ floats $= 4$\,MB each); effective bandwidth $\approx 300$--$500$\,GB/s aggregate.
\end{itemize}
Consequently, any algorithm that cannot reuse data in the L1/L2 caches will hit either the $45 \text{ GB/s}$ DRAM wall or the softer L3-bandwidth ceiling, severely bottlenecking operations like BLAS~1 and BLAS~2.

\paragraph{Software Environment.}
\begin{itemize}
  \item \textbf{Compiler}: GCC~13.3, utilizing aggressive optimization flags: \texttt{-O3 -march=native -mavx2 -mfma -fopenmp}.
  \item \textbf{Reference}: OpenBLAS (system package version), executed with identical \texttt{OMP\_NUM\_THREADS} to ensure a fair comparison.
  \item \textbf{Timing Methodology}: To combat OS jitter and CPU frequency scaling noise, all benchmarks employ an adaptive timing loop. A kernel is executed repeatedly until the coefficient of variation (CV) of the runtimes falls below 2\%, and the median time is reported.
\end{itemize}

\section{SGEMM development}

\subsection{BLIS framework}

Matrix multiplication $C = \alpha A B + \beta C$ requires $O(N^3)$ operations but only $O(N^2)$ data. By carefully orchestrating memory access, we can achieve an arithmetic intensity (FLOPs per byte transferred) high enough to keep the AVX2 ALUs constantly fed without stalling on main memory.

We adopted the BLIS architecture, which decomposes the GEMM operation into five nested loops surrounding a highly optimized hardware-specific micro-kernel. 
\begin{enumerate}
    \item \textbf{Loop 5 (JC)}: Slices $N$ into blocks of size $NC$.
    \item \textbf{Loop 4 (PC)}: Slices $K$ into blocks of size $KC$. Matrix $B$ is packed into a contiguous L3-resident buffer.
    \item \textbf{Loop 3 (IC)}: Slices $M$ into blocks of size $MC$. Matrix $A$ is packed into a contiguous L2-resident buffer.
    \item \textbf{Loop 2 (JR)}: Steps through the $NC$ block in chunks of $NR$.
    \item \textbf{Loop 1 (IR)}: Steps through the $MC$ block in chunks of $MR$.
\end{enumerate}
At the very center lies the Micro-Kernel, computing an $MR \times NR$ block of $C$ by taking the dot product of an $MR \times KC$ panel of $A$ and a $KC \times NR$ panel of $B$.

\subsection{Week~1 8$\times$8 AVX2 micro-kernel}

The AI was provided with the BLIS algorithmic blueprint and tasked to generate the baseline SGEMM. 
It \textbf{succeeded} in producing a functionally correct 5-loop framework. For the micro-kernel, it selected an $MR=8, NR=8$ configuration.

\paragraph{Register blocking.}
To compute an $8 \times 8$ block of $C$, we need to store 64 single-precision floats. Since one 256-bit YMM register holds 8 floats, the $C$ accumulator block requires $64 / 8 = 8$ YMM registers.
During the inner $K$-loop, we must load a column of $A$ (1 register) and broadcast elements of $B$ (1 register). This requires $8 \text{ (accumulators)} + 1 \text{ (A)} + 1 \text{ (B)} = 10$ registers, fitting comfortably within the 16 YMM registers available in AVX2.

Single-thread performance on a $1024^3$ matrix reached $\sim$40\,GF/s (71\% of the 56\,GF/s core peak). While impressive for a first iteration, it fell short of OpenBLAS's $\sim$50\,GF/s, indicating register underutilization.

\subsection{Week~2 micro-kernels and parallelism}

\paragraph{Register-bound micro-kernels (6$\times$16 and 4$\times$24).}
To improve the ratio of memory loads to arithmetic operations, the AI was instructed to maximize the use of the 16 YMM registers. Two new kernels were successfully generated:
\begin{itemize}
  \item \textbf{6$\times$16 Micro-kernel}: $MR=6$ output \emph{rows}, $NR=16$ output columns. For each $k$-step, each of the 6 rows of $A$ contributes a \emph{scalar broadcast} (one YMM register, reused across the $NR/8=2$ FMA chains); $B$ is loaded as $NR/8 = 2$ YMM vectors. The output accumulator block requires $MR \times (NR/8) = 6 \times 2 = 12$ YMM registers. Total register pressure: $12 \text{ (C accumulators)} + 2 \text{ (B)} + 1 \text{ (A broadcast)} = 15$ YMM, leaving 1 free. Each $k$-step performs $6 \times 2 = 12$ FMAs ($= 24$\,FLOPs).
  \item \textbf{4$\times$24 Micro-kernel}: $MR=4$ output rows, $NR=24$ columns. Each of the 4 rows of $A$ is a scalar broadcast; $B$ requires $NR/8 = 3$ YMM loads. Accumulator block: $4 \times 3 = 12$ registers. Total: $12 \text{ (C)} + 3 \text{ (B)} + 1 \text{ (A broadcast)} = 16$ YMM registers, perfectly saturating the AVX2 register file with zero register spills.
\end{itemize}
Both configurations were generated \textbf{correctly on the first try}, complete with pointer arithmetic and inline-assembly equivalents.

\paragraph{2D task parallelism.}
Initial attempts used \texttt{omp taskloop collapse(2)}, which introduced 10$\times$ overhead due to task spawn/join costs (see \S\ref{sec:parallel}). This was replaced with loop-based parallelism.
It was replaced by \texttt{omp parallel for collapse(2) schedule(dynamic)}.

\paragraph{K replication.}
The K dimension is split into $r$ slices, each writing a private partial-$C$
buffer, then merged via a binary reduction tree. The structure was generated
correctly by AI; the initial \emph{sequential} reduction was a separate bug fixed
in Week~3.

\subsection{Week~3 micro-kernel optimization}

\subsubsection{Loop unrolling $\times$4}
Extending the unroll factor from $\times$2 to $\times$4 increased arithmetic throughput by reducing loop overhead per FMA. This yielded 2--8\% speedup for medium and large matrix sizes.

\subsubsection{Software prefetching}
Prefetch hints were placed within the 4-step unrolled loop to reduce L1 miss penalties. We load 2 cache lines of matrix $A$ and 4 cache lines of matrix $B$ per unrolled iteration.

\begin{lstlisting}[caption={6$\times$16 prefetch pattern inside the $\times$4 loop}]
_mm_prefetch((const char*)(pfA),      _MM_HINT_T0); // A CL0
_mm_prefetch((const char*)(pfA + 16), _MM_HINT_T0); // A CL1
pfA += 4 * MR;   // 96 bytes = 2 CLs ahead
_mm_prefetch((const char*)(pfB),      _MM_HINT_T0); // B CL0..CL3
_mm_prefetch((const char*)(pfB + 16), _MM_HINT_T0);
_mm_prefetch((const char*)(pfB + 32), _MM_HINT_T0);
_mm_prefetch((const char*)(pfB + 48), _MM_HINT_T0);
pfB += 4 * NR;   // 256 bytes = 4 CLs ahead
\end{lstlisting}

Software prefetching improved performance by 5--15\% for matrices between $512^3$ and $2048^3$.

\subsubsection{Non-temporal stores}
Non-temporal stores bypass the L3 cache and write directly to memory, reducing cache pollution. This requires 32-byte alignment, which we verify before applying the optimization:
\begin{lstlisting}
if (use_nt_store && ((uintptr_t)Cr % 32 == 0))
    _mm256_stream_ps(Cr, result);     // bypass L3 write
else
    _mm256_storeu_ps(Cr, result);
\end{lstlisting}
Non-temporal stores improve throughput by 5--20\% for large matrices ($2048^3$) but provide no benefit for small matrices where the L3 is underutilized.

\subsubsection{Assembly constraints}
When implementing the $6\times16$ micro-kernel in inline assembly, we encountered GCC's 30-operand limit for extended inline assembly. Rather than passing k4 (k\_len/4) as a named operand, we compute it directly inside the assembly block:
\begin{lstlisting}
"movl   %[klen], %%eax\n\t"
"shrl   $2, %%eax\n\t"     // eax = k_len / 4
\end{lstlisting}

\subsubsection{Structural fixes}

Three known Week~2 limitations were addressed. Figure~\ref{fig:optim_steps}
shows the GFLOP/s at each step.

\begin{itemize}
  \item \textbf{Parallel reduction}: We replaced the sequential final reduction with a recursive binary reduction tree using OpenMP task dependencies. For $1024^3$ matrices, this recovered 1.52 GF/s (8.48 to 10.0 GF/s, +18\%).
  \item \textbf{Small-matrix bypass}: Matrices with $MNK < 128^3$ are computed without tiling overhead. This improved $64^3$ performance from 34.1 to 42.3 GF/s (+24\%).
  \item \textbf{Adaptive blocking parameter}: When the block parameter MC satisfies $M/MC < 2 \cdot \text{threads}$, we reduce MC to increase parallelism. At 20 threads on $128^3$, this raised performance from 75.3 to 126.6 GF/s (+68\%).
\end{itemize}

\section{SGEMM parallel scheduling}
\label{sec:parallel}
%%============================================================

In Week 3, we successfully distributed SGEMM across multiple cores. In Week 4, we undertook a rigorous comparative analysis of \textbf{how} that work is distributed, comparing traditional Loop scheduling with modern Task scheduling models.

\subsection{Loop-based parallelism}
Loop-based parallelism statically or dynamically slices a nested set of \texttt{for} loops and assigns exact ranges of iterations to specific threads. 
\textit{Analogy}: An established assembly line where every worker knows exactly which widgets (tiles of matrix $C$) they are responsible for building before the shift even begins.
Because the plan is made upfront and the iteration space is perfectly regular (a 2D grid), the OpenMP runtime overhead is virtually zero. 

\subsection{Task-based parallelism}
Task-based parallelism treats every iteration (every tile of matrix $C$) as an independent "Task". A single master thread spawns these tasks, packing all the required variables (\texttt{firstprivate}) into a descriptor block in memory, and pushes them into a queue. Any idle thread can pop a task and execute it.
\textit{Analogy}: A ticket-based kitchen line. When an order (tile) comes in, a ticket is printed. Any available chef can grab the next ticket and cook it. It is incredibly flexible for unbalanced workloads but requires time to print and read the tickets.

\paragraph{Small matrices ($N \le 256$).}
For a $128 \times 128$ matrix, the work required to multiply a $6 \times 16$ tile is infinitesimal (measured in instructions). The time it takes OpenMP to allocate a block of memory, capture the local variables, push the task to a queue, and have a worker thread pop it is orders of magnitude larger than the math itself. \textbf{Task overhead completely dominates execution time.}

\paragraph{Large matrices ($N \ge 1024$).}
Matrix multiplication scales as $O(N^3)$ for computation, but only $O(N^2)$ for the number of grid tiles (tasks). Therefore, as the matrix grows, the time spent actually doing math inside the micro-kernel dwarfs the initial time spent printing the "task tickets". By $N=2048$, the overhead difference shrinks to merely $\sim$3.7\%.

\subsection{SSYRK}

\paragraph{Iteration 1.}
AI delegated to \texttt{sgemm\_ex(ib, jb, k, alpha, A\_i, lda,
\textbf{A\_j}, lda, ...)} without realizing \texttt{sgemm\_ex} expects a
$K\times jb$ matrix as the second argument, but $A_j$ is $jb\times K$.
All tests failed (max error $\approx 0.4$ vs tolerance $10^{-3}$).
\textbf{AI did not detect the argument layout mismatch.}

\paragraph{Iteration 2.}
Each output element $C_{ii,jj} = \alpha \sum_k A_{ii,k}A_{jj,k}$ is computed
as an AVX2 dot product of two contiguous rows:
\begin{lstlisting}
const float *Ai = a + ii * lda, *Aj = a + jj * lda;
__m256 acc = _mm256_setzero_ps();
for (int kk = 0; kk <= k - 8; kk += 8)
    acc = _mm256_fmadd_ps(_mm256_loadu_ps(Ai+kk),
                          _mm256_loadu_ps(Aj+kk), acc);
// horizontal sum -> c[ii*ldc+jj] = alpha*s + beta*c[...]
\end{lstlisting}
128$\times$128 tiling for outer-loop L2 reuse; OMP dynamic parallel over triangular
tiles. \textbf{All 24 tests pass.} Throughput $\sim$4~GF/s ($n=256$, no panel
packing, deferred).

\subsection{STRSM and STRMM}

Both generated correctly by AI on the first attempt. STRSM uses scalar
back-substitution over columns of $B$ (the only parallelizable dimension);
STRMM uses a buffer-copy approach. \textbf{Correctness challenge}: the AI-generated
test generator used uniform $U[0.5,1.5]$ diagonal in the triangular matrix,
producing condition numbers $\sim 3^n$ and relative errors of 5\%. Manual fix:
\texttt{diagonal = 2.0 + rand()}, \texttt{off-diagonal = 0.01*rand()}.

\paragraph{Why we are slower.}

\textbf{SSYRK.}
Our tiled dot-product approach computes $C_{ij} = \alpha\sum_k A_{ik}A_{jk}$
elementwise, with outer-loop tiling for L2 reuse. It avoids packing but
accesses $A$ column-wise for the second argument, causing cache-line
inefficiency for large $k$. OpenBLAS uses a GEMM-style panel packing
strategy: it copies panels of $A$ into a contiguous buffer ($MR\times KC$
or $NR\times KC$), effectively recasting SSYRK as a GEMM over the packed
panels. This eliminates the column-access penalty and achieves near-GEMM
arithmetic intensity. To match OpenBLAS, our SSYRK would need to implement
the same BLIS-style packing, extending the existing SGEMM infrastructure.

\textbf{STRSM.}
The triangular solve $B := \alpha\, A^{-1} B$ is inherently sequential: column
$j$ of the solution depends on all previously solved columns $0, \ldots, j-1$.
Our scalar back-substitution processes one column of $B$ at a time, exploiting
parallelism only in the independent row updates within each column
(\texttt{omp parallel for} over rows). OpenBLAS uses a blocked algorithm that
first applies a TRSM on a small diagonal block (fitting in registers), then
updates the remaining columns with a GEMM call, amortising the serial
dependency cost across large blocks. Reaching parity would require implementing
this right-looking blocked TRSM.

\textbf{STRMM.}
Our buffer-copy approach copies the triangular matrix $A$ into a dense buffer
before calling SGEMM, incurring $O(N^2)$ copy overhead that dominates at small
$N$. OpenBLAS fuses the triangular structure directly into its packing routine,
adding zero copy overhead. Replacing our dense-copy approach with a
triangularity-aware pack would eliminate this bottleneck.

\section{Peak utilization}
%%============================================================

SGEMM is compute-bound (not bandwidth-bound) due to BLIS panel reuse. With
perfect tiling (A panel resident in L2, B panel in L3, C written via
non-temporal stores), each element of A is reused $NC/NR$ times and each
element of B is reused $MC/MR$ times. Dominant DRAM traffic reduces to one
pass of A and one pass of B:
\begin{equation}
  \mathrm{AI} = \frac{2N^3}{(N^2 + N^2)\times 4\;\mathrm{B/float}}
              = \frac{N}{4}\;\mathrm{FLOP/byte}
              \approx 256\;\mathrm{FLOP/byte} \quad (N{=}1024)
  \label{eq:sgemm_ai}
\end{equation}
This far exceeds the roofline crossover ($\approx 1.3$\,FLOP/byte at
56\,GF/s peak and 45\,GB/s DDR4), confirming SGEMM is firmly in the
compute-bound regime for all practical matrix sizes.

\section{BLAS~1 implementation strategy}

To explore BLAS~1 optimization, we undertook development of 9 vector kernels (\texttt{sscal}, \texttt{scopy}, \texttt{sswap}, \texttt{saxpy}, \texttt{sdot}, \texttt{snrm2}, \texttt{sasum}, \texttt{isamax}, \texttt{srot}). A key observation from prior HPC work is that BLAS~1 behaves fundamentally differently from BLAS~3: each is memory-bandwidth-bound with $O(1)$ arithmetic per byte, making vectorization and parallelization effects unpredictable without measurement.

We asked the AI to generate all 9 kernels in parallel, using a staged approach: scalar baseline, then AVX2 vectorization, then OpenMP parallelism. The working hypothesis was that by comparing each stage, we could identify where each technique contributes and where it causes overhead.

\paragraph{Implementation plan.}
To enable reproducible generation of BLAS~1 kernels from scratch, we documented the following algorithmic strategy:

\begin{enumerate}
    \item \textbf{Classify by Operation Type}: Partition kernels into three classes:
    \begin{itemize}
        \item \textbf{Bandwidth-Limited} (sscal, scopy, sswap, saxpy): Perform $\sim$1 FLOP per byte; already saturate L3 bandwidth at single thread.
        \item \textbf{Latency-Limited Reductions} (sdot, snrm2, sasum): Accumulate results via long dependency chains; ALU stalls on FMA latency ($\sim$5 cycles).
        \item \textbf{Index-Based} (isamax): Search for maximum; data-dependent branching prevents vectorization.
    \end{itemize}
    \item \textbf{Bandwidth-Limited}: Use contiguous memory access ($\texttt{incx} = 1$) with AVX2 unrolling. Do NOT parallelize; single thread already saturates bandwidth. For strided access ($\texttt{incx} > 1$), use gather intrinsics or scalar fallback.
    \item \textbf{Latency-Limited}: Use 4--8 independent accumulators (depends on register availability) to hide FMA latency. Parallelize with \texttt{\#pragma omp parallel for} after vectorization, because scalar ALU is idle waiting for results.
    \item \textbf{Index-Based}: Accept scalar implementation; vectorization not applicable for data-dependent branches like \texttt{isamax}.
    \item \textbf{Reduction Across Threads}: For multi-threaded reductions (e.g., parallel \texttt{sdot}), use atomic operations or work-stealing to minimize synchronization overhead.
\end{enumerate}

\paragraph{Stage 1 results.}
The AI generated scalar and AVX2 variants on the first iteration. We measured the AVX2 impact: for reduction operations (\texttt{sdot}, \texttt{snrm2}, \texttt{sasum}), results showed 4.3$\times$ to 8.8$\times$ speedup. This gain was expected because the AI used multiple independent accumulators (\texttt{\_\_m256}) to allow FMA pipeline to fill during the long dependency chain. Interestingly, bandwidth-limited operations (\texttt{sscal}, \texttt{saxpy}) showed 0--1$\times$ speedup, suggesting that single-threaded core bandwidth was already saturated and wider SIMD did not help.

\paragraph{Stage 1 issue.}
However, when we tested non-unit strides (\texttt{incx > 1}), the AI defaulted to scalar loops, eliminating vectorization gains. We hypothesized that \texttt{\_mm256\_i32gather\_ps} could restore SIMD, but benchmarking revealed gather itself was slow (serial on our CPU). Instead, we proposed that the AI use gather only for very large strides, and unrolled scalar loops for modest strides (2--8). After guiding the AI with this constraint, performance recovered to 1.5--2$\times$ vs scalar.

\paragraph{Stage 2 issue.}
When we asked the AI to parallelize reductions with \texttt{\#pragma omp} for 8 threads, the initial code used \texttt{\#pragma omp critical} on the accumulator. Benchmarking showed this caused severe lock contention. We diagnosed this and reformulated: thread-local accumulators followed by a sequential final reduction. This design achieved approximately 4$\times$ better scaling, nearly saturating 8-thread speedup.

\subsection*{Stage 3 multi-threading}
We then tested the hypothesis: if single-threaded cores are L3-bandwidth-saturated, can we exploit the fact that L3 is shared among 8 cores? The idea is that L3 has much higher internal bandwidth than the DDR4 interface, and multiple threads reading from L3 might cooperatively saturate this internal bandwidth.

Results confirmed this hypothesis dramatically: \texttt{sscal} reaches 134.4\,GB/s (7$\times$ single-threaded), \texttt{sdot} reaches 54.6\,GFLOP/s. By comparison, OpenBLAS remains at 18.1 and 7.5 respectively. Analysis suggests OpenBLAS deliberately avoids BLAS~1 threading for typical use (small vectors where thread overhead matters), while our large-vector workload ($N=10^6$ in 25\,MB L3) can amortize thread start/join costs. 

To interpret these numbers relative to hardware: peak per-core is $2 \text{ FMA/cycle} \times 3.5\,\text{GHz} = 56\,\text{GFLOP/s}$. Single-threaded \texttt{sdot} reaches 7.3\,GFLOP/s (13\% of peak), confirming the bottleneck is memory bandwidth, not compute. Multi-threaded \texttt{sdot} at 54.6\,GFLOP/s redistributes the pressure onto aggregate L3 bandwidth.

\section{BLAS~2 algorithms and architecture}

BLAS~2 operations require $O(N^2)$ memory access and $O(N^2)$ arithmetic. At arithmetic intensity 1 (roughly 2 FLOPs per 4 bytes loaded), these kernels are memory-bound. We hypothesized that algorithmic restructuring could overcome this bottleneck where naive implementations do not.

\paragraph{\texttt{ssyr2} fusion.}
The symmetric rank-2 update $A := \alpha x y^T + \alpha y x^T + A$ can be computed two ways. The naive approach calls rank-1 updates twice, reading and writing $A$ twice ($4 \times O(N^2)$ bandwidth). 

We instead fuse both updates into a single pass. For each row $i$, we load $\alpha x_i$ and $\alpha y_i$ into broadcast registers, then perform dual FMA operations on the loaded row of $A$. This cuts memory traffic to $2 \times O(N^2)$.

\begin{lstlisting}[caption={ssyr2 Fused AVX2 Inner Loop}]
__m256 v_x_i = _mm256_set1_ps(alpha * x[i]);
__m256 v_y_i = _mm256_set1_ps(alpha * y[i]);

for (int j = 0; j <= i - 8; j += 8) {
    __m256 v_A = _mm256_loadu_ps(&A[i * lda + j]);
    __m256 v_y = _mm256_loadu_ps(&y[j]);
    __m256 v_x = _mm256_loadu_ps(&x[j]);
    
    // Dual FMA fusion: A = A + (alpha*x_i)*y + (alpha*y_i)*x
    v_A = _mm256_fmadd_ps(v_x_i, v_y, v_A);
    v_A = _mm256_fmadd_ps(v_y_i, v_x, v_A);
    
    _mm256_storeu_ps(&A[i * lda + j], v_A);
}
\end{lstlisting}

This halved the total memory traffic to $2 \times O(N^2)$. As shown in Figure~\ref{fig:blas2_plots}, this change led to \textbf{1.2$\times$ to 2.2$\times$ speedup over OpenBLAS} across the tested matrix sizes.

\paragraph{\texttt{sger} limit.}
For the general rank-1 update \texttt{sger} ($A = \alpha x y^T + A$), the implementation uses row-streaming access. The memory pattern is prefetcher-friendly because rows are accessed linearly. At matrix sizes $N \ge 512$, single-threaded performance reaches 38--39\,GB/s, which matches the bandwidth of the dual-channel DDR4 interface. This indicates both our implementation and OpenBLAS reach the same hardware ceiling (main memory), leaving no room for algorithmic improvement.
\paragraph{\texttt{sgemv} transpose.}
For the transpose case \texttt{sgemv} ($y = \alpha A^T x + \beta y$), the initial implementation used a column-major loop:
\begin{lstlisting}[caption={Naive Transpose Loop}]
for (int i = 0; i < n; i++)
    for (int j = 0; j < m; j++)
        y[i] += A[j * lda + i] * x[j];
\end{lstlisting}
Measurement showed 1.7\,GB/s (versus OpenBLAS's 32\,GB/s). Root cause analysis: C matrices are row-major, so incrementing \texttt{j} jumps by $lda$ bytes (4KB for $N=1024$). Every iteration fetches a new cache line and uses $\sim$4 bytes, wasting 60 bytes per line. 

When we asked the AI to parallelize this directly, it applied \texttt{\#pragma omp} to the outer loop. This introduced a secondary problem: false sharing on output vector $y$—multiple threads writing adjacent elements in the same 64-byte line caused cache coherence overhead. 

Reformulation: We proposed to rewrite the transposed form as a row-wise \texttt{saxpy}-like operation where each thread owns a slice of $y$ and iterates sequentially over rows. This eliminated both column-stride misses and false sharing.

\paragraph{\texttt{ssymv} limit.}
\textbf{Iteration 1.} AI generated a loop over the stored lower triangle with a conditional
\texttt{if (j <= i)} guard. The branching prevented auto-vectorisation; single-thread throughput
was 0.8\,GB/s.
\textbf{Iteration 2.} We separated the diagonal, lower-triangle, and reflected (upper) accesses
into three independent loops, enabling AVX2 on the lower part. Performance improved to
1.6\,GB/s — still 14$\times$ below OpenBLAS (22\,GB/s). Root cause: the reflected part accesses
column $j$ for all rows $i < j$, which strides by $\texttt{lda}$ bytes — each iteration fetches a new cache line for one float. This is the same column-stride pathology as \texttt{sgemv} Transpose.
Fully resolving this requires a blocked panel layout: copying the symmetric matrix into a
row-major packed panel so that the reflected accesses become row-wise.
OpenBLAS implements exactly this strategy (BLIS-style packing), achieving near-bandwidth
utilisation. We accepted the 0.07$\times$ result as a documented architectural limitation.

\paragraph{OpenBLAS comparison.}
While we beat OpenBLAS in \texttt{ssyr2} and matched it in \texttt{sger}, OpenBLAS heavily outperforms us in \texttt{sgemv} Transpose and \texttt{ssymv}. The reason is cache-blocking: OpenBLAS uses assembly-level panel loops that fetch a block of matrix $A$ into the L2 cache, then compute multiple elements of the output simultaneously. Our \texttt{\#pragma omp parallel for} processes one row at a time. \section{Testing and numerical stability}

Validating high-performance floating-point math requires far more than simple equality checks. To ensure the unquestionable correctness of all BLAS~1, 2, and 3 kernels against edge cases and numerical instability, we developed a sophisticated automated C testbench.

\paragraph{Uniform random matrices.}
The most common mistake in testing linear algebra software is populating matrices with uniform random numbers, $U(-1, 1)$. For operations like triangular solvers (\texttt{strsm, strsv}), a uniformly populated matrix has a condition number $\kappa(A)$ that grows exponentially with the matrix dimension $N$. 
In floating-point arithmetic, the relative error of the output is bounded by $\epsilon \cdot \kappa(A)$, where $\epsilon$ is machine precision ($\approx 10^{-7}$ for FP32). For $N=512$, $\kappa(A)$ easily exceeds $10^6$, causing the solver to amplify tiny roundoff errors into massive macroscopic differences, causing completely valid code to fail the tests.

\paragraph{Matrix conditioning.}
To solve this, our testbench utilizes specialized data generators:
\begin{itemize}
    \item \textbf{Positive-Definiteness (\texttt{rand\_symm})}: For symmetric operations, we generate diagonally dominant symmetric matrices. We populate the matrix with $U(0,1)$, mirror it, and then add $N$ to the diagonal. This guarantees positive eigenvalues and absolute numerical stability.
    \item \textbf{Well-Conditioned Triangular (\texttt{rand\_tri})}: For solvers, we restrict off-diagonal elements to $0.01 \cdot U(0,1)$, while forcing the main diagonal to $2.0 + U(0,1)$. This restricts the condition number to $O(1)$, completely isolating algorithm flaws from mathematical artifacts.
\end{itemize}

\paragraph{Parallel reduction tolerance.}
Floating-point addition is not associative: $(a+b)+c \neq a+(b+c)$. Because our OpenMP implementations divide matrices across cores and accumulate partial sums in an unpredictable order, our results will legitimately differ from the sequential OpenBLAS reference.
To handle this, the testbench enforces a strict relative tolerance check:
$$ \frac{|x_{\text{ours}} - x_{\text{ref}}|}{\max(|x_{\text{ours}}|, |x_{\text{ref}}|) + 10^{-6}} \le \text{tol} $$
For standard operations, $\text{tol} = 10^{-3}$. For deep reduction operations like \texttt{ssyrk} where thousands of products are accumulated in parallel, the tolerance is safely relaxed to $5\times10^{-3}$. All 1287 BLAS 1 and 118 BLAS 2/3 test cases pass under these conditions.

\section{Agent loop}

A cornerstone of this project was the development of the ``Agent Loop'':
two complementary Python scripts that together form an autonomous
compile-benchmark-optimise cycle.
\texttt{sgemm\_tuner.py} implements the three black-box search algorithms
described in \S3 (Gradient Descent, Bayesian Optimisation, Simulated
Annealing); it treats the SGEMM kernel as a pure function from block-size
parameters $(\text{MC}, \text{KC}, \text{NC})$ to measured GF/s.
\texttt{agent\_loop.py} wraps \texttt{sgemm\_tuner.py} into an end-to-end
pipeline: it writes a C header with the chosen parameters, invokes
\texttt{make}, runs the benchmark binary, parses the output, and feeds the
result back to the tuner --- all without human intervention. This is the
``agentic'' aspect: the human sets a budget (number of evaluations) and the
loop runs until convergence or budget exhaustion.

The loop was designed to compile, benchmark, and optimize C code by navigating the hardware parameter space.

\paragraph{Search space and updates.}
The loop explored critical micro-architectural variables that are notoriously difficult for humans to manually tune:
\begin{itemize}
    \item \textbf{Loop Unrolling Factors}: Testing 1$\times$, 2$\times$, 4$\times$, and 8$\times$ inner-loop unrolling to balance instruction cache sizes against pipeline depth.
    \item \textbf{Software Prefetch Distances}: Tuning the \texttt{\_MM\_HINT\_T0} distances (0, 32, 64, 128 bytes) to fetch data into L1 cache exactly when the ALUs need it, compensating for DDR4 latency.
    \item \textbf{Cache Blocking (\texttt{MC, KC, NC})}: Sizing the BLIS macro-blocks to ensure panels fit precisely within the $32$\,KB L1, $256$\,KB L2, and shared $25$\,MB L3 caches without suffering eviction.
\end{itemize}

\paragraph{Bayesian optimization.}
In Week 3, a Bayesian Optimizer was deployed to tune SGEMM block sizes. The optimizer builds a Gaussian Process surrogate model of the performance landscape and selects configurations by maximizing Expected Improvement (EI), balancing exploration of unknown parameters with exploitation of fast configurations.

The optimization discovered non-power-of-two block sizes (\texttt{MC=480, KC=256, NC=10432}) that are difficult to find by manual tuning, achieving 4\% speedup over mathematically derived GotoBLAS defaults.

\paragraph{Black-box limits.}
The agent loop also showed where parameter tuning is insufficient. For structural bottlenecks, such as $O(\mu s)$ task creation overhead for microsecond-scale \texttt{taskloop} work or column-stride misses in \texttt{ssymv}, changing prefetch distance or unroll factors has little effect. In practice, this creates a near-flat search landscape for methods such as Simulated Annealing.

\paragraph{Overall effect.}
The autonomous agent loop was responsible for recovering the final \textbf{10--15\% of hardware performance}. It acted as a crucial polishing tool, pushing algorithmically sound AVX2 code to the ultimate ceiling of the silicon (reaching 104\%--117\% of the un-tuned baselines). However, the foundational 85\% of the performance gains were strictly the domain of intelligent, AI-guided architectural redesigns: fusing loops, recasting memory access patterns, and selecting lightweight synchronization primitives.

\section{AI-assisted development}

\subsection*{What worked}

The BLIS 5-loop framework (correct loop nesting, packing buffers, pointer arithmetic)
came out right on the first attempt and compiled without modification.
Reaching the same point by reading the Goto and Van Zee papers alone
would have taken at least two days; with the AI it took an afternoon.
The 6$\times$16 and 4$\times$24 micro-kernels were also correct on the first try,
including the register pressure analysis.

For reduction kernels (\texttt{sdot}, \texttt{snrm2}, \texttt{sasum}), the AI
independently used four independent accumulators without being asked.
This is the right technique for hiding 5-cycle FMA latency, and the AI had
apparently internalized it from context.

The \texttt{ssyr2} fusion was the most useful AI-generated insight.
When asked to minimise memory traffic, the AI proposed collapsing two rank-1
passes into one, halving the number of reads and writes to $A$.
The suggestion required understanding both the mathematical identity
$\alpha xy^T + \alpha yx^T = $ single loop and how to express it with dual FMAs.
The result was a 2.2$\times$ speedup over OpenBLAS.

The testbench also came out well: diagonally-dominant matrix construction, relative
tolerance for parallel reductions, clear error messages.
These are the parts of numerical testing that are easy to get wrong,
and the AI got them right.

\subsection*{Where it failed}

The recurring failure mode was that the AI could not predict hardware behavior
before seeing a benchmark.

\textit{Task parallelism.}
The AI proposed \texttt{omp taskloop collapse(2)} for GEMM tile dispatch.
At $N=128^3$, this was 14$\times$ slower than \texttt{parallel for}.
The reason: each tile takes $<$10\,$\mu$s, task creation overhead is $\sim$1\,$\mu$s,
and there are hundreds of tiles per matrix.
When we showed the benchmark result, the AI suggested increasing tile size.
We switched to loop parallelism and the problem disappeared.

\textit{Column-stride access in sgemv Transpose.}
The AI wrote the mathematically natural formulation:
iterate over columns of $A^T$, accumulate into $y$.
In a row-major layout, this strides by \texttt{lda} bytes per inner iteration,
fetching a new 64-byte cache line for every 4 bytes used.
Measured throughput: 1.7\,GB/s versus OpenBLAS's 32\,GB/s.
The AI's diagnosis was "try prefetching the next column."
The actual fix was rewriting the algorithm: iterate over rows of $A$
and accumulate partial contributions to all of $y$ via saxpy-style updates.
Once we described this, the AI implemented it correctly.
The lesson is not that the AI failed, but that it cannot make this diagnosis
from a performance number alone.

\textit{False sharing.}
After reformulating sgemv Transpose, we asked the AI to parallelise it with
\texttt{omp parallel for} over the outer loop.
Performance did not improve.
The cause: threads writing adjacent elements of $y$ in the same 64-byte cache
line, triggering coherence invalidation.
The fix was assigning contiguous chunks of $y$ to each thread
(private output ranges, no shared cache lines).
The AI implemented this once we identified the issue; it could not identify it
from the symptoms.

\textit{Assembly operand limits.}
GCC's extended inline assembly supports at most 30 named operands.
The AI generated a 32-operand block for the 6$\times$16 kernel.
The fix was computing the loop counter inside the asm block with a shift
instruction rather than passing it as a named constraint.
This is not in any standard reference, and the AI did not know the limit
until the compiler reported it.

\subsection*{What would have helped}

Two changes would have compressed the iteration cycles significantly.

First: direct benchmark access.
Each iteration took 30--60 seconds on our end (compile, run, read output,
summarise, prompt).
The AI could not observe the output directly.
For performance debugging, this meant describing symptoms in text rather
than sharing a hardware counter trace.
If the AI had been able to compile and run the benchmark autonomously,
most of the failures above would have been caught in minutes rather than hours.

Second: hardware latency models.
The AI has no internalized model of DDR4 access latency ($\sim$60\,ns),
L3 ring bandwidth as a function of active core count, or NUMA first-touch
allocation policy.
Every time we said "this access pattern will miss caches on this machine,"
the AI could produce a correct fix.
But it could not say that unprompted.
A lightweight hardware model, even an approximate one, would have caught
the column-stride and taskloop failures before any code was written.

\subsection*{Reflections}

Looking at the six weeks overall, the AI shortened the scaffolding phase
(framework generation, SIMD boilerplate, test infrastructure) by roughly half.
For the optimization phase, it was more useful as an executor than a diagnostician:
we identified what needed to change, the AI wrote the code.

The concern about passive learning is real but nuanced.
The AI-generated code that just worked taught us less than the code that failed.
The taskloop overhead, column-stride pathology, and false sharing are things
we now understand precisely because we had to trace them by hand.
In that sense the failures were not a cost of using AI; they were the part
of the project with the highest educational density.

\section{Conclusion}

We built a working BLAS library over six weeks using AI-assisted development
and measured its performance against OpenBLAS across 19 kernels.
SGEMM at 20 threads beats OpenBLAS by 1.3--2.5$\times$ for medium-to-large matrices,
largely because our adaptive tile-count balancing distributes work more evenly
across 20 cores than the system OpenBLAS build.
BLAS~1 kernels on large cache-resident vectors outperform single-threaded OpenBLAS
by large factors, though this comparison is not entirely fair since OpenBLAS
does not thread BLAS-1 by default.
BLAS~2 showed both an algorithmic win (\texttt{ssyr2}, 2.2$\times$ via fusion)
and a clear architectural ceiling (\texttt{ssymv}, 0.07$\times$ due to
column-stride cache misses we did not resolve).

The consistent pattern across all three levels was this:
the AI produced correct code quickly when the algorithm was well-specified,
and produced plausible-but-wrong code when the bottleneck was hardware-dependent.
Closing the gap between "plausible" and "fast" required measurement, diagnosis,
and reformulation, all done by hand.
The agent loop recovered the final 10--15\% once the algorithms were structurally sound,
which was the right division of labour: black-box optimization cannot fix a
fundamentally wrong access pattern, and manual tuning cannot search a 5D parameter
space in a reasonable time.
