\section*{Introduction}
\addcontentsline{toc}{section}{Introduction}
%%============================================================

This report documents the design, implementation, and rigorous evaluation of a
high-performance single-precision Basic Linear Algebra Subprograms (BLAS) library. 
Developed over a six-week period, this project serves as a case study in AI-driven 
High-Performance Computing (HPC) development. The entire codebase, testing infrastructure, 
and autotuning loops were generated through iterative prompting and guided architectural 
decisions using the Antigravity~IDE (powered by advanced LLMs such as Claude~Sonnet~4.6 and Gemini~Pro).

The ultimate objective is not just functional correctness, but reaching the absolute 
hardware limits of a modern multi-core workstation. The library is benchmarked against 
OpenBLAS, the industry-standard highly optimized reference implementation, to provide 
a realistic assessment of our AI-generated kernels.

\subsection*{Scope of Development}
The project spans six weeks of systematic, iterative development, categorized as follows:
\begin{itemize}
  \item \textbf{Part~1 (Weeks 1--3 \& 6): GEMM and BLAS~3.} The core focus is SGEMM (Single-Precision General Matrix Multiply), implementing the sophisticated BLIS (BLAS-like Library Instantiation Software) 5-loop architecture. This part also covers autotuning infrastructures and derived BLAS~3 operations (\texttt{ssyrk, strsm, strmm}).
  \item \textbf{Part~2 (Week 4): BLAS~1.} Implementation of foundational vector-vector operations (\texttt{sscal, scopy, sswap, saxpy, sdot, snrm2, sasum, isamax, srot}), pushing single-thread and multi-thread memory bandwidth limits.
  \item \textbf{Part~3 (Week 5): BLAS~2.} Implementation of matrix-vector operations (\texttt{sgemv, ssymv, strmv, strsv, sger, ssyr, ssyr2}), focusing on cache-friendly access patterns and algorithmic optimizations like loop fusion.
  \item \textbf{Part~4: Testing and Agentic Optimization.} Development of the rigorous numerical validation testbench and the autonomous parameter-tuning agent loop.
\end{itemize}

\subsection*{Hardware Architecture and Theoretical Limits}
Understanding the target hardware is critical for evaluating HPC software. The testbed is a 20-core Intel Xeon workstation (x86-64 architecture).

\paragraph{Compute Capabilities (AVX2 \& FMA).}
The processor supports Advanced Vector Extensions 2 (AVX2), featuring 256-bit wide vector registers. A single 256-bit register holds 8 single-precision floating-point numbers. The processor also supports Fused Multiply-Add (FMA) instructions, which perform $d = a \cdot b + c$ in a single hardware step, counting as 2 floating-point operations (FLOPs). 
A single core can issue two FMA instructions per clock cycle. At a boost frequency of 3.5\,GHz, the theoretical peak performance is:
\begin{equation}
\text{Peak}_{\text{core}} = 2 \text{ (instructions/cycle)} \times 8 \text{ (elements/vector)} \times 2 \text{ (FLOPs/FMA)} \times 3.5\text{e}9 \text{ (cycles/sec)} = 56 \text{ GFLOP/s}
\end{equation}
Fully utilized across all 20 cores, the machine theoretical peak is $1120 \text{ GFLOP/s}$ (1.12 TFLOP/s).

\paragraph{Memory Hierarchy and Bandwidth.}
The memory subsystem is equally critical. The CPU features three levels of cache:
\begin{itemize}
    \item \textbf{L1 Data Cache}: 32\,KB per core. Extremely fast (4-5 cycles), but small.
    \item \textbf{L2 Cache}: 256\,KB per core. Intermediate latency ($\sim$12-14 cycles).
    \item \textbf{L3 Cache}: $\approx$25\,MB, shared across all cores.
\end{itemize}
The main memory (RAM) utilizes DDR4-3200 modules. The theoretical maximum memory bandwidth for a single memory controller channel is roughly $25.6 \text{ GB/s}$. With a dual-channel configuration, the practical sustained bandwidth is approximately $45 \text{ GB/s}$.
Consequently, any algorithm that cannot reuse data in the L1/L2 caches will hit the $45 \text{ GB/s}$ "memory wall," severely bottlenecking operations like BLAS~1 and BLAS~2.

\paragraph{Software Environment.}
\begin{itemize}
  \item \textbf{Compiler}: GCC~13.3, utilizing aggressive optimization flags: \texttt{-O3 -march=native -mavx2 -mfma -fopenmp}.
  \item \textbf{Reference}: OpenBLAS (system package version), executed with identical \texttt{OMP\_NUM\_THREADS} to ensure a fair comparison.
  \item \textbf{Timing Methodology}: To combat OS jitter and CPU frequency scaling noise, all benchmarks employ an adaptive timing loop. A kernel is executed repeatedly until the coefficient of variation (CV) of the runtimes falls below 2\%, and the median time is reported.
\end{itemize}

\section{SGEMM Development}

\subsection{The BLIS Architectural Framework}

Matrix multiplication $C = \alpha A B + \beta C$ requires $O(N^3)$ operations but only $O(N^2)$ data. By carefully orchestrating memory access, we can achieve an arithmetic intensity (FLOPs per byte transferred) high enough to keep the AVX2 ALUs constantly fed without stalling on main memory.

We adopted the BLIS architecture, which decomposes the GEMM operation into five nested loops surrounding a highly optimized hardware-specific micro-kernel. 
\begin{enumerate}
    \item \textbf{Loop 5 (JC)}: Slices $N$ into blocks of size $NC$.
    \item \textbf{Loop 4 (PC)}: Slices $K$ into blocks of size $KC$. Matrix $B$ is packed into a contiguous L3-resident buffer.
    \item \textbf{Loop 3 (IC)}: Slices $M$ into blocks of size $MC$. Matrix $A$ is packed into a contiguous L2-resident buffer.
    \item \textbf{Loop 2 (JR)}: Steps through the $NC$ block in chunks of $NR$.
    \item \textbf{Loop 1 (IR)}: Steps through the $MC$ block in chunks of $MR$.
\end{enumerate}
At the very center lies the Micro-Kernel, computing an $MR \times NR$ block of $C$ by taking the dot product of an $MR \times KC$ panel of $A$ and a $KC \times NR$ panel of $B$.

\subsection{Week~1 -- The 8$\times$8 AVX2 Micro-kernel}

The AI was provided with the BLIS algorithmic blueprint and tasked to generate the baseline SGEMM. 
It \textbf{succeeded} in producing a functionally correct 5-loop framework. For the micro-kernel, it selected an $MR=8, NR=8$ configuration.

\paragraph{Micro-kernel Register Blocking.}
To compute an $8 \times 8$ block of $C$, we need to store 64 single-precision floats. Since one 256-bit YMM register holds 8 floats, the $C$ accumulator block requires $64 / 8 = 8$ YMM registers.
During the inner $K$-loop, we must load a column of $A$ (1 register) and broadcast elements of $B$ (1 register). This requires $8 \text{ (accumulators)} + 1 \text{ (A)} + 1 \text{ (B)} = 10$ registers, fitting comfortably within the 16 YMM registers available in AVX2.

Single-thread performance on a $1024^3$ matrix reached $\sim$40\,GF/s (71\% of the 56\,GF/s core peak). While impressive for a first iteration, it fell short of OpenBLAS's $\sim$50\,GF/s, indicating register underutilization.

\subsection{Week~2 -- Advanced Micro-kernels and Parallelism}

\paragraph{Pushing Register Limits (6$\times$16 and 4$\times$24).}
To improve the ratio of memory loads to arithmetic operations, the AI was instructed to maximize the use of the 16 YMM registers. Two new kernels were successfully generated:
\begin{itemize}
  \item \textbf{6$\times$16 Micro-kernel}: We cannot divide $M=6$ evenly into 8-float registers. However, we can treat $M=6$ as $1$ full register (with 2 floats masked/ignored). For $N=16$, we need 2 vectors. The $C$ block requires $1 \text{ (M-vec)} \times 2 \text{ (N-vecs)} \times 6 \text{ (unroll)} = 12$ accumulator registers. We need 2 registers for $B$ and 1 for $A$, totaling 15 YMM registers. This leaves 1 register free and performs 12 FMAs per $K$-step (192 FLOPs per unrolled step).
  \item \textbf{4$\times$24 Micro-kernel}: $M=4$ uses a half-register (via \texttt{\_mm\_load\_ps} or masking). $N=24$ requires 3 vectors. Accumulators: $1 \times 3 \times 4 = 12$ registers. Loading $B$ requires 3 registers, and $A$ requires 1. Total: $16$ YMM registers. This perfectly saturates the AVX2 register file.
\end{itemize}
Both configurations were generated \textbf{correctly on the first try}, complete with pointer arithmetic and inline-assembly equivalents.

\paragraph{2D task parallelism (TASK1).}
The AI initially chose \texttt{omp taskloop collapse(2)} --- a catastrophic
mistake caught by benchmarking (10$\times$ overhead; see \S\ref{sec:parallel}).
It was replaced by \texttt{omp parallel for collapse(2) schedule(dynamic)}.

\paragraph{3D K-replication (TASK2).}
The K dimension is split into $r$ slices, each writing a private partial-$C$
buffer, then merged via a binary reduction tree. The structure was generated
correctly by AI; the initial \emph{sequential} reduction was a separate bug fixed
in Week~3.

\subsection{Week~3 -- Micro-kernel Optimization}

\subsubsection{Loop Unrolling $\times$4}
All C and ASM kernels were extended from $\times$2 to $\times$4.
\textbf{AI succeeded immediately.} Benefit: 2--8\% at medium/large sizes.

\subsubsection{Software Prefetching}
Next-tile prefetch hints spread across the 4-step loop. The AI placed them
correctly on the first attempt, loading 2~CLs of~A and 4~CLs of~B per
4-step iteration.

\begin{lstlisting}[caption={6$\times$16 prefetch pattern inside the $\times$4 loop}]
_mm_prefetch((const char*)(pfA),      _MM_HINT_T0); // A CL0
_mm_prefetch((const char*)(pfA + 16), _MM_HINT_T0); // A CL1
pfA += 4 * MR;   // 96 bytes = 2 CLs ahead
_mm_prefetch((const char*)(pfB),      _MM_HINT_T0); // B CL0..CL3
_mm_prefetch((const char*)(pfB + 16), _MM_HINT_T0);
_mm_prefetch((const char*)(pfB + 32), _MM_HINT_T0);
_mm_prefetch((const char*)(pfB + 48), _MM_HINT_T0);
pfB += 4 * NR;   // 256 bytes = 4 CLs ahead
\end{lstlisting}

Measured benefit: +5--15\% for $512^3$--$2048^3$.

\subsubsection{Non-Temporal Stores for C}
AI generated \texttt{\_mm256\_stream\_ps} store path but \textbf{omitted the
32-byte alignment check} --- caught during code review. Manual fix:
\begin{lstlisting}
if (use_nt_store && ((uintptr_t)Cr % 32 == 0))
    _mm256_stream_ps(Cr, result);     // bypass L3 write
else
    _mm256_storeu_ps(Cr, result);
\end{lstlisting}
Benefit: +5--20\% for $2048^3$; neutral for small sizes.

\subsubsection{ASM Kernel Bug (AI Failure)}
The ASM kernel failed to compile: \texttt{error: more than 30 operands in `asm'}.
The AI tried to pass \texttt{k4 = k\_len/4} as a named operand, hitting GCC's
30-operand limit. \textbf{Manual fix}: compute k4 inside the assembly block:
\begin{lstlisting}
"movl   %[klen], %%eax\n\t"
"shrl   $2, %%eax\n\t"     // eax = k_len / 4
\end{lstlisting}

\subsubsection{Structural Fixes and Their Impact}

Three known Week~2 limitations were addressed. Figure~\ref{fig:optim_steps}
shows the GFLOP/s at each step.

\begin{itemize}
  \item \textbf{Parallel TASK2 reduction}: sequential $\to$ recursive binary tree
        (OMP task dependencies). AI succeeded. $1024^3$: 8.48 $\to$ 10.0~GF/s (+18\%).
  \item \textbf{Small-matrix bypass}: skip tiling for $MNK < 128^3$.
        AI succeeded. $64^3$: 34.1 $\to$ 42.3~GF/s (+24\%).

        AI succeeded. $128^3$ at 20T: 75.3 $\to$ 126.6~GF/s (+68\%).
\end{itemize}

\section{SGEMM Parallel Scheduling Analysis: Loops vs. Tasks}
\label{sec:parallel}
%%============================================================

In Week 3, we successfully distributed SGEMM across multiple cores. In Week 4, we undertook a rigorous comparative analysis of \textbf{how} that work is distributed, comparing traditional Loop scheduling with modern Task scheduling models.

\subsection{Concept 1: Loop-Based Parallelism (\texttt{omp for collapse})}
Loop-based parallelism statically or dynamically slices a nested set of \texttt{for} loops and assigns exact ranges of iterations to specific threads. 
\textit{Analogy}: An established assembly line where every worker knows exactly which widgets (tiles of matrix $C$) they are responsible for building before the shift even begins.
Because the plan is made upfront and the iteration space is perfectly regular (a 2D grid), the OpenMP runtime overhead is virtually zero. 

\subsection{Concept 2: Task-Based Parallelism (\texttt{omp task} / \texttt{omp taskloop})}
Task-based parallelism treats every iteration (every tile of matrix $C$) as an independent "Task". A single master thread spawns these tasks, packing all the required variables (\texttt{firstprivate}) into a descriptor block in memory, and pushes them into a queue. Any idle thread can pop a task and execute it.
\textit{Analogy}: A ticket-based kitchen line. When an order (tile) comes in, a ticket is printed. Any available chef can grab the next ticket and cook it. It is incredibly flexible for unbalanced workloads but requires time to print and read the tickets.

\paragraph{The Small Matrix Disaster ($N \le 256$):} 
For a $128 \times 128$ matrix, the work required to multiply a $6 \times 16$ tile is infinitesimal (measured in instructions). The time it takes OpenMP to allocate a block of memory, capture the local variables, push the task to a queue, and have a worker thread pop it is orders of magnitude larger than the math itself. \textbf{Task overhead completely dominates execution time.}

\paragraph{The Convergence ($N \ge 1024$):}
Matrix multiplication scales as $O(N^3)$ for computation, but only $O(N^2)$ for the number of grid tiles (tasks). Therefore, as the matrix grows, the time spent actually doing math inside the micro-kernel dwarfs the initial time spent printing the "task tickets". By $N=2048$, the overhead difference shrinks to merely $\sim$3.7\%.

\section{Implementation Strategy and AI Generation}

The BLAS~1 development involved creating 9 vector kernels (\texttt{sscal}, \texttt{scopy}, \texttt{sswap}, \texttt{saxpy}, \texttt{sdot}, \texttt{snrm2}, \texttt{sasum}, \texttt{isamax}, \texttt{srot}). The AI was tasked to generate all 9 kernels simultaneously, following an incremental methodology: starting with a simple scalar baseline, vectorizing with AVX2, and finally parallelizing with OpenMP.

\paragraph{AI Successes.}
The AI successfully generated the scalar and basic AVX2 versions for all 9 kernels on the first attempt. For reduction operations (\texttt{sdot}, \texttt{snrm2}, \texttt{sasum}), the AI correctly utilized multiple independent accumulators (\texttt{\_\_m256}) to hide FMA instruction latency. This correctly prevented pipeline stalls, resulting in an immediate 4.3$\times$ to 8.8$\times$ speedup over the scalar baselines.

\paragraph{AI Failures and Iteration.}
The AI initially failed to properly vectorize operations with non-unit strides (\texttt{incx > 1}). It opted to use a scalar fallback for all non-unit strides, severely limiting performance for strided access.
\textbf{Manual Intervention:} The AI was instructed to optimize strided memory access. It successfully self-corrected by utilizing the \texttt{\_mm256\_i32gather\_ps} intrinsic, which restored AVX2 utilization and provided up to a 1.5$\times$--2$\times$ speedup over the scalar fallback for strides between 2 and 8.

\section{Performance Analysis and Deep Dive}

To accurately measure the impact of each optimization technique, we developed the kernels in three progressive stages, and benchmarked them thoroughly against OpenBLAS.

\subsection*{Concept 1: The Scalar Baseline (The "Naive" Approach)}
The scalar baseline represents the most straightforward way to write the algorithm in C using a simple \texttt{for} loop, processing one element at a time. It relies entirely on the compiler and the CPU's default mechanisms to handle memory and computation.
As expected, processing one element per clock cycle fails to fully utilize the CPU's capabilities. For memory-bound operations, it slowly fetches data. For compute-bound operations (like \texttt{snrm2} or \texttt{sdot}), it leaves the arithmetic logic units (ALUs) starved.

\subsection*{Concept 2: SIMD Vectorization (AVX2)}
Modern CPUs feature wide execution lanes known as SIMD (Single Instruction, Multiple Data). Using AVX2 intrinsics, we manually rewrote the kernels to load 8 floating-point numbers into a 256-bit register, perform the arithmetic (e.g., 8 multiplications at once using FMA), and store 8 results back to memory in a single instruction sequence. 

\begin{lstlisting}[caption={AVX2 sdot inner loop with 4 independent accumulators to hide latency}]
__m256 sum0 = _mm256_setzero_ps();
__m256 sum1 = _mm256_setzero_ps();
__m256 sum2 = _mm256_setzero_ps();
__m256 sum3 = _mm256_setzero_ps();

for (int i = 0; i <= n - 32; i += 32) {
    sum0 = _mm256_fmadd_ps(_mm256_loadu_ps(&x[i]),    _mm256_loadu_ps(&y[i]),    sum0);
    sum1 = _mm256_fmadd_ps(_mm256_loadu_ps(&x[i+8]),  _mm256_loadu_ps(&y[i+8]),  sum1);
    sum2 = _mm256_fmadd_ps(_mm256_loadu_ps(&x[i+16]), _mm256_loadu_ps(&y[i+16]), sum2);
    sum3 = _mm256_fmadd_ps(_mm256_loadu_ps(&x[i+24]), _mm256_loadu_ps(&y[i+24]), sum3);
}
// Sum reduction follows...
\end{lstlisting}

\paragraph{Why Our Multi-Threading Defeats OpenBLAS.}
When OpenMP threading is layered on top of AVX2, our implementation experiences a massive leap: \texttt{sscal} throughput jumps to 134.4\,GB/s, and \texttt{sdot} climbs to 54.6\,GFLOP/s. 
By comparison, OpenBLAS stagnates at 18.1\,GB/s and 7.5\,GFLOP/s. This staggering discrepancy reveals a deliberate architectural choice within OpenBLAS: it typically refuses to spawn threads for Level 1 BLAS operations, assuming the overhead of waking thread pools outweighs the math for simple vectors. However, when processing $N=10^6$ elements residing entirely in the 25\,MB L3 cache, a single thread is utterly incapable of draining the L3 cache's massive internal bandwidth. Our explicit \texttt{\#pragma omp parallel for schedule(static)} efficiently divides the vector among 8 cores, allowing all 8 L1 caches to pull from the L3 cache simultaneously.

\paragraph{Machine Peak Computations.}

Our single-threaded \texttt{sdot} achieves 19.7\,GFLOP/s (35\% of theoretical max). With 8 threads, the combined system peak is 448\,GFLOP/s. Our 8-thread implementation hits 56.4\,GFLOP/s. We have shifted the bottleneck from single-core ALU throughput to multi-core L3 cache bandwidth.

\section{AI-Driven Development: Algorithms vs. Architecture}

BLAS~2 operations involve multiplying matrices by vectors, requiring $O(N^2)$ memory reads and $O(N^2)$ arithmetic operations. Because the arithmetic intensity is strictly $O(1)$ (approximately 2 FLOPs per 4 bytes loaded), these kernels are entirely dominated by memory bandwidth and cache-line utilization. 

\paragraph{Success: Algorithmic Fusion in \texttt{ssyr2}.}
The symmetric rank-2 update is defined as $A = \alpha x y^T + \alpha y x^T + A$. 
The AI initially proposed an inefficient two-pass approach, calling the rank-1 update (\texttt{ssyr}) twice in sequence. This meant the matrix $A$ was read from memory, modified, and written back to memory \textit{twice}, consuming $4 \times O(N^2)$ bandwidth.

When explicitly prompted to optimize for memory bandwidth, the AI successfully conceptualized an algorithmic fusion. It merged both updates into a single loop pass. By maintaining two broadcast registers for $\alpha x_i$ and $\alpha y_i$, it applied dual FMA instructions to a single vector loaded from $A$, before writing it back once.

\begin{lstlisting}[caption={ssyr2 Fused AVX2 Inner Loop}]
__m256 v_x_i = _mm256_set1_ps(alpha * x[i]);
__m256 v_y_i = _mm256_set1_ps(alpha * y[i]);

for (int j = 0; j <= i - 8; j += 8) {
    __m256 v_A = _mm256_loadu_ps(&A[i * lda + j]);
    __m256 v_y = _mm256_loadu_ps(&y[j]);
    __m256 v_x = _mm256_loadu_ps(&x[j]);
    
    // Dual FMA fusion: A = A + (alpha*x_i)*y + (alpha*y_i)*x
    v_A = _mm256_fmadd_ps(v_x_i, v_y, v_A);
    v_A = _mm256_fmadd_ps(v_y_i, v_x, v_A);
    
    _mm256_storeu_ps(&A[i * lda + j], v_A);
}
\end{lstlisting}

This halved the total memory traffic to $2 \times O(N^2)$. As shown in Figure~\ref{fig:blas2_plots}, this profound algorithmic shift resulted in our implementation running \textbf{1.2$\times$ to 2.2$\times$ faster than OpenBLAS} across all tested matrix sizes.

\paragraph{Success: Bandwidth Saturation in \texttt{sger}.}
For the general rank-1 update (\texttt{sger}: $A = \alpha x y^T + A$), the AI seamlessly generated a tight AVX2 FMA row-update loop. The memory access pattern (streaming linearly through rows of $A$) is highly hardware-prefetcher friendly. At $N \ge 512$, our 1-thread implementation reached 38--39\,GB/s, effectively saturating the dual-channel DDR4 memory bus and perfectly mirroring OpenBLAS.

\paragraph{The Catastrophe of Column-Major Access: \texttt{sgemv} Transpose.}
The transposition required by \texttt{sgemv} ($y = \alpha A^T x + \beta y$) exposed the limitations of AI when dealing with low-level hardware intricacies. The AI intuitively wrote the mathematical equivalent:
\begin{lstlisting}[caption={Naive sgemv Transpose (Cache Thrashing)}]
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        y[i] += A[j * lda + i] * x[j];
    }
}
\end{lstlisting}
While mathematically sound, this code causes catastrophic L1 cache thrashing. Matrices in C are row-major. Incrementing the row index \texttt{j} implies leaping over $lda$ elements (e.g., 4\,KB for $N=1024$). The CPU pulls an entire 64-byte cache line from RAM to read a single 4-byte float, then evicts it immediately on the next iteration. This resulted in a dismal 1.7\,GB/s throughput, compared to OpenBLAS's 32\,GB/s (a $0.05\times$ ratio).

When asked to parallelize this, the AI applied OpenMP to the outer loop. This caused a secondary hardware disaster: \textbf{False Sharing}. Multiple cores were simultaneously updating adjacent floats within the same 64-byte cache line of the $y$ vector, causing the L1 cache coherence protocol to endlessly bounce the line between cores. 

\textbf{Manual Intervention:} The AI required explicit human guidance to recast the transposition as a row-wise \texttt{saxpy} operation distributed over threads. By assigning each thread a slice of the output vector $y$, and iterating contiguously across the rows of $A$, we restored sequential memory access and full AVX2 vectorization.

\paragraph{Failure: \texttt{ssymv} Branching.}
For \texttt{ssymv}, the AI initially used a branched access \texttt{min(i,j)} to read the symmetric matrix, which entirely prevented vectorization. When instructed to split the loop into stored and reflected halves, performance barely improved (1.6\,GB/s vs OB 22\,GB/s) because the reflected half still suffered from column-stride cache misses. Resolving this would require a blocked panel scatter approach, which the AI could not independently derive.

\section{Testing Methodology: Numerical Stability}

Validating high-performance floating-point math requires far more than simple equality checks. To ensure the unquestionable correctness of all BLAS~1, 2, and 3 kernels against edge cases and numerical instability, we developed a sophisticated automated C testbench.

\paragraph{The Danger of Uniform Random Matrices.}
The most common mistake in testing linear algebra software is populating matrices with uniform random numbers, $U(-1, 1)$. For operations like triangular solvers (\texttt{strsm, strsv}), a uniformly populated matrix has a condition number $\kappa(A)$ that grows exponentially with the matrix dimension $N$. 
In floating-point arithmetic, the relative error of the output is bounded by $\epsilon \cdot \kappa(A)$, where $\epsilon$ is machine precision ($\approx 10^{-7}$ for FP32). For $N=512$, $\kappa(A)$ easily exceeds $10^6$, causing the solver to amplify tiny roundoff errors into massive macroscopic differences, causing completely valid code to fail the tests.

\paragraph{Engineered Matrix Conditioning.}
To solve this, our testbench utilizes specialized data generators:
\begin{itemize}
    \item \textbf{Positive-Definiteness (\texttt{rand\_symm})}: For symmetric operations, we generate diagonally dominant symmetric matrices. We populate the matrix with $U(0,1)$, mirror it, and then add $N$ to the diagonal. This guarantees positive eigenvalues and absolute numerical stability.
    \item \textbf{Well-Conditioned Triangular (\texttt{rand\_tri})}: For solvers, we restrict off-diagonal elements to $0.01 \cdot U(0,1)$, while forcing the main diagonal to $2.0 + U(0,1)$. This restricts the condition number to $O(1)$, completely isolating algorithm flaws from mathematical artifacts.
\end{itemize}

\paragraph{Handling Non-Associative Multi-Threading.}
Floating-point addition is not associative: $(a+b)+c \neq a+(b+c)$. Because our OpenMP implementations divide matrices across cores and accumulate partial sums in an unpredictable order, our results will legitimately differ from the sequential OpenBLAS reference.
To handle this, the testbench enforces a strict relative tolerance check:
$$ \frac{|x_{\text{ours}} - x_{\text{ref}}|}{\max(|x_{\text{ours}}|, |x_{\text{ref}}|) + 10^{-6}} \le \text{tol} $$
For standard operations, $\text{tol} = 10^{-3}$. For deep reduction operations like \texttt{ssyrk} where thousands of products are accumulated in parallel, the tolerance is safely relaxed to $5\times10^{-3}$. All 1287 BLAS 1 and 118 BLAS 2/3 test cases pass under these conditions.

\section{The Autonomous Agent Loop Architecture}

A cornerstone of this project was the development of the "Agent Loop" (\texttt{agent\_loop.py} and \texttt{sgemm\_tuner.py})---an autonomous Python framework designed to iteratively compile, benchmark, and optimize C code by directly navigating the multidimensional hardware parameter space.

\paragraph{Search Space and Mechanisms.}
The loop explored critical micro-architectural variables that are notoriously difficult for humans to manually tune:
\begin{itemize}
    \item \textbf{Loop Unrolling Factors}: Testing 1$\times$, 2$\times$, 4$\times$, and 8$\times$ inner-loop unrolling to balance instruction cache sizes against pipeline depth.
    \item \textbf{Software Prefetch Distances}: Tuning the \texttt{\_MM\_HINT\_T0} distances (0, 32, 64, 128 bytes) to fetch data into L1 cache exactly when the ALUs need it, compensating for DDR4 latency.
    \item \textbf{Cache Blocking (\texttt{MC, KC, NC})}: Sizing the BLIS macro-blocks to ensure panels fit precisely within the $32$\,KB L1, $256$\,KB L2, and shared $25$\,MB L3 caches without suffering eviction.
\end{itemize}

\paragraph{Success: Bayesian Optimization.}
During Week 3, we deployed a sophisticated Bayesian Optimizer to tune the SGEMM block sizes. Unlike random search, the Bayesian Optimizer builds a probabilistic surrogate model (a Gaussian Process) of the performance landscape. It selects the next configuration to test by maximizing the \textit{Expected Improvement} (EI) acquisition function, inherently balancing the exploration of unknown parameters against the exploitation of known fast parameters.
This rigorous mathematical approach successfully navigated a flat, noisy performance landscape to find non-obvious, non-power-of-two optimal block sizes (\texttt{MC=480, KC=256, NC=10432}). This autonomous discovery secured a 4\% speedup over the mathematically derived GotoBLAS defaults.

\paragraph{Limitations of Black-Box Optimization.}
The agent loop revealed the absolute limits of parameter tuning. When faced with fundamental structural flaws---such as the $O(\mu s)$ overhead of spawning \texttt{\#pragma omp taskloop} tasks for microsecond-scale work, or the column-stride cache thrashing in \texttt{ssymv}---no amount of parameter tweaking could salvage the performance. Black-box optimizers like Simulated Annealing flailed hopelessly because modifying a prefetch distance on a severely bottlenecked algorithm yields zero measurable difference, creating a perfectly flat gradient.

\paragraph{Overall Efficacy Conclusion.}
The autonomous agent loop was responsible for recovering the final \textbf{10--15\% of hardware performance}. It acted as a crucial polishing tool, pushing algorithmically sound AVX2 code to the ultimate ceiling of the silicon (reaching 104\%--117\% of the un-tuned baselines). However, the foundational 85\% of the performance gains were strictly the domain of intelligent, AI-guided architectural redesigns: fusing loops, recasting memory access patterns, and selecting lightweight synchronization primitives.
